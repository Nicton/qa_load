jmeter.report.generator.bat

@pushd "%~dp0"
@setlocal
set JVM_ARGS=-Xms300m -Xmx2g -DfileEncoding=UTF-8

D:\Tools\apache-jmeter-5.1\bin\jmeter.bat ^
--reportonly "%1" ^
--addprop jmeter.report.generator.bat.properties ^
--reportoutputfolder "%1.report"

@popd

jmeter.report.generator.bat.properties

# Defines the overall granularity for over time graphs
# Granularity must be higher than 1000 (1second) otherwise Throughput graphs will be incorrect
# see Bug 60149
jmeter.reportgenerator.overall_granularity=10000


# Response Time Distribution graph definition
jmeter.reportgenerator.graph.responseTimeDistribution.classname=org.apache.jmeter.report.processor.graph.impl.ResponseTimeDistributionGraphConsumer
jmeter.reportgenerator.graph.responseTimeDistribution.title=Response Time Distribution
jmeter.reportgenerator.graph.responseTimeDistribution.property.set_granularity=500

jpgc.repo.sendstats=false

Создайте два таких файла. Настройки - путь к JMeter (нужно поменять), настройки памяти для генератора (Xms, Xmx), частота формирования точек на графиках (гранулярность) задайте по своему усмотрению.

Как пользоваться:
1. Настроить в JMeter запись статистики в csv/jtl файл.
2. Перенести с помощью мыши файл со статистикой на иконку файла jmeter.report.generator.bat.
3. Посмотреть отчёт в каталоге *.report рядом с логом csv/jtl.
4. Если точек получилось слишком много или мало, то скорректировать гранулярность в файле properties и повторить 2.

===========
Обычный тест - на поиск максимальной производительности:

- составляется список критериев - как должна работать система:
(Например: Вход в систему должен отрабатывать не более, чем за ХХ секунд, допустимый процент ошибок не более YY).

- сценарий составляется так, что бы нагрузка увеличивалась ступеньками

- Исследуются результаты: на какой ступеньке любой из критериев нарушился.

- соответственно, ступенька с максимальной нагрузкой, где критерии соблюдаются и есть максимальная производительность системы.
